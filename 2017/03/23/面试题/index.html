    1. 中讯面试题

    JVM内存模型

    程序计数器：

    1. 很小的一块内存空间，只要用来记录字节码指令执行的位置
    2. 每个线程都有自己的程序计数器
    3. 做多线程切换时，程序计数器可以用来记录上次执行的字节码指令位置

    虚拟机栈

    1. 在内存模型中主要用来描述方法的调用
    2. 每个方法的执行都会创建一个栈帧，用来存储局部变量表、操作数栈、动态链接、方法出入口等信息
    3. 方法的调用过程，对应栈帧的入栈出栈过程
    4. 一般此区域会产生两种异常：
      1. StackOverflowError（线程请求的栈深度大于虚拟机允许的深度）
      2. OutOfMemoryError （如果虚拟机可以动态扩展，如果无法申请到足够内存）

    本地方法栈

    1. 和虚拟机栈的作用相似
    2. 不同点是虚拟机栈是记录虚拟机执行方法的过程，本地方法栈是记录native本地方法执行的过程

    堆

    1. heap是OOM故障的发源地
    2. 存储着几乎所有的实例变量
    3. 由垃圾回收器自动回收管理
    4. 各个子线程共享使用
    5. 在内存模型中占用的空间也是最大的
    6. 堆内存大小可以设置成固定的也可以设置成动态调整的（-Xms设定初始值、-Xmx设定最大值）

    方法区

    1. 所有线程共享使用的区域
    2. 用来存储被虚拟机加载的类信息、常量、静态变量、JIT（just in time即时编译技术）编译后的代码等数据。
    3. 运行时常量池也是方法区的一部分，用来存放编译时期生成的各种字面量和符号引用

    B+树和B树的区别

    1. B树的特点
      1. B 树本质上一棵平衡树（不是平衡二叉树）
      2. 每个节点上可以存储多个元素
      3. 每个节点（非叶子节点）既存指针也存数据
      4. 每个叶子节点到根节点的距离相同
    2. B树的添加
    3. B树的删除
    4. B+树的特点
    5. B+树的添加
    6. B+树的删除
    7. 总结
      1. B+树的非叶子节点不存储数据，那么可以存储更多的数据，使得树的高度更矮，查询速度更快
      2. B+树的叶子节点之间通过链表与相邻节点相连，插入与删除更加方便快捷，无需通过父节点移动元素
      3. B+树的查询能力更加稳定，数据都在叶子节点，高度相同（B树的查询能力就不那么稳定了，因为非叶子节点也存储了数据）

    为什么使用线程池

    1. 为什么使用线程池
      1. 我们平时使用Java开发，是支持多线程运行运行的，而线程的生命周期又包括：创建、就绪、运行、阻塞、销毁等过程，线程的创建与销毁是非常损耗性能的，假如我们每个请求任务都对应一个线程的话，对系统将会是极大的开销（线程的创建与销毁、CPU的上下文切换、内存溢出等情况），并且若是无限制的开辟线程处理任务，也会导致OOM的发生。
    2. 线程池的原理
      1. 线程池是线程的一个集合
      2. 需要使用线程的时候，之间从线程池获取，省去了线程的创建与销毁过程
      3. 初始化线程池时会根据配置的核心线程数，初始化一些核心线程
      4. 当任务数量大于核心线程数量时，就会把任务放置到阻塞队列中
      5. 当阻塞队列满时，会再开辟新线程来出来刚添加的任务（直到达到最大线程数量）
      6. 线程达到最大线程数量时，则开始进行拒接策略（直接拒绝并抛异常、开辟新线程处理、忽略此任务、删除一个最老的任务并把当前任务添加到队列）
    3. 线程池核心参数
      1. corePoolSize 核心线程数
      2. maxPoolSize 最大线程数
      3. queueCapacity 阻塞队列workQueue的大小
      4. keepAliveTime 线程空闲最大等待时间（回收等待时间）
      5. allowCoreThreadTimeout 运行核心线程超时（设置为true时，核心线程达到空闲时间临界值也会回收）
      6. rejectedExecutionHandler 任务拒绝处理器
    4. 线程池的种类
      1. newFixedThreadPool(int n) 获取固定数量的线程池
      2. newCachedThreadPool() 获取动态数量的线程池，如果不够则创建新的，无上限
      3. newSingThreadExecutor() 获取单个线程的线程池，只有一个线程
      4. newScheduledThreadPool() 创建固定大小的线程池，可以延时周期执行定时任务

    分布式系统主要解决哪些问题

    1. 分布式解决的问题
      1. 解决了扩展问题，一般传企业都是采用垂直扩展，通过扩充CPU、内存、磁盘来提高该机器的处理能力，但是这种垂直扩展是有限度的，而且是昂贵的。分布式方案可以通过搭建集群，把廉价的机器组合在一起，统一进行资源调度，来提高任务的处理能力。
      2. 解决了高可用问题
      3. 降低了数据的延时性（通过策略，访问就近的集群）
    2. 分布式的主要挑战
      1. 数据一致性问题

    数据库索引

    1. 索引概述
      1. 索引是为了提高数据库访问速度而产生的
      2. 底层采用的是一种平衡树的数据结构（B树、B+树）
    2. 索引特点
      1. 避免了全表扫描，只扫描较少的索引页和数据页，而不是查询所有的数据页
      2. 聚集索引可以避免数据的插入操作，集中于表的最后一个数据页（排序操作）
      3. 在某些情况下，索引可以避免排序操作
    3. 聚集索引
      1. 没加主键的表，在磁盘上的存储是杂乱无章的
      2. 加了主键的表，在磁盘上的存储是一排排的整整齐齐
      3. 既是给表加了主键之后，整个表就变成了一个索引，也就是所谓的聚集索引
      4. 一张表只能又一个主键、一个聚集索引，因为主键的作用就是把表的数据结构转化为索引（平衡树）的数据格式
      5. 聚集索引的所有节点都是由主键字段中的数据构成的，通常是主键的ID字段
      6. 在数据库里面，所有行都会按照主键索引进行排序
    4. 非聚集索引
      1. 非聚集索引和聚集索引一样， 同样是采用平衡树作为索引的数据结构
      2. 给普通字段上添加索引
      3. 非聚集索引的所有节点都是存储的主键值，想要通过非聚集索引查询数据的话，需要二次查询（先查出主键值，再通过聚集索引查询数据）
    5. 覆盖索引
      1. 可以避免非聚集索引的二次查询
      2. 查找字段刚好是创建的复合索引，可以直接查出数据

    搞懂select/poll/epoll

    参考：搞懂 select/poll/epoll，就这篇了 - 知乎 (zhihu.com)

    1. select特点
      1. 一个线程管理众多的socket链接
      2. 调用select的时候，会把所有管理的socket的fd传到内核中
      3. socket会有属于自己的睡眠队列，select安排一个内应，即被管理的socket睡眠队列里面塞一个entry（当socket接收到网卡数据之后，就会去它的睡眠队列中遍历entry, 调用entry的callback方法唤醒select）
      4. 被唤醒的select并不知道是那个socket有数据了，仍然会遍历所有socket
      5. select的限制，因为socket fd需要从用户空间copy到内核空间，为了控制拷贝的大小而做了限制，即是每个select能copy的fds集合大小只有1024
    2. poll的特点
      1. 优化了fds的存储接口，select是有的是bit数组，poll使用了pollfd（可以解决1024的限制）
    3. epoll的特点
      1. epoll做了两点优化
        1. 无需像select复制fds到内核，新增或删除socket的时候就调用epoll_ctl，直接在内核中管理socket集合
        2. 唤醒之后，无需像select那也遍历所有的socket，直接遍历ready_list链表（callback 里面会把当前的 socket 加入到 ready_list 然后唤醒 epoll）
      2. ET与LT
        1. ET，边沿触发
          1. 照上面的逻辑就是 epoll 遍历 ready_list 的时候，会把 socket 从 ready_list 里面移除，然后读取这个 scoket 的事件
        2. LT，水平触发
          1. 在这个模式下 epoll 遍历 ready_list 的时候，会把 socket 从 ready_list 里面移除，然后读取这个 scoket 的事件，如果这个 socket 返回了感兴趣的事件，那么当前这个 socket 会再被加入到 ready_list 中，这样下次调用 epoll_wait 的时候，还能拿到这个 socket。

    Mysql相关

    参考：https://mp.weixin.qq.com/s/qKGUiglWF0EWcnFOlU3JBA

    1. Mysql体系结构

      1. 可以分为两大块：server与存储引擎

      2. SQl语句的执行流程

        1. 客户端与Server建立链接后，一条Sql语句经过TCP从客户端传输到Server
        2. Server对Sql语句进行词法、语法分析（词法分析器）
        3. 语法没问题之后，通过优化器对Sql进行重写（如何选择驱动表、如何选择合适的索引等），目的是让Sql可以更高效的执行（我们平时使用explain查看的执行计划，其实就是优化器帮我们优化后的执行方案）
        4. 优化之后，就由执行器调用存储引擎接口来执行语句了

      3. 关键点

        例如我现在要执行一条select * from yes where name='yes的练级攻略';这条语句，name 这一列没有索引。

        此时流程如下：

        1. Server 调用存储引擎的返回这个表的第一行这个接口，此时 Server 拿到第一行数据。
        2. Server 通过 where 条件判断 name 是否等于yes的练级攻略，如果是则放到结果集中，不是则跳过。
        3. Server 继续调用存储引擎的接口来下一行！，然后再通过 where 条件来判断。
        4. 如此循环往复，直到最后一行记录。
        5. 不会等结果全部收集完毕了才返回给客户端，等集满net_buffer大小的结果就会发送，也就是边查边发。

        思考：

        ​ 1.若是Mysql不是采用边查边发的方案，那么多查询都要缓存全部数据，数据库内存早就撑爆了

        ​ 2.查询条件不是索引的时候，返回的是这个表的第一行数据，若是索引的话，返回的是符合条件的第一行数据（因此使用索引可以更快的查询出正确的数据）

    2. InnoDB与MyISAM存储引擎

      1. Mysql 5.5.8之前，一直使用的是MyISAM存储引擎

      2. MyISAM存储引擎特点

        1. 不支持事务
        2. 不支持行级锁，只支持表锁
        3. 适用于OLAP的场景（读多写少、查询分析类的场景）
        4. 它的数据和索引是分离的（不在一个文件上存储），并且数据库只会缓存索引数据，数据文件由操作系统缓存
        5. MyISAM存储引擎也使用的是B+树，只是不像InnoDB一样叶子节点存储完整数据，它的节点只存储索引，所以对它来说主键索引与非主键索引没什么区别

      3. InnoDB存储引擎的特点

        1. 支持事务，实现了四种隔离级别的事务，利用MVCC来支持高并发
        2. 默认的事务隔离级别是RR可重复读
        3. 支持行锁，间隙锁
        4. 利用行锁+间隙锁，可以避免幻读的能力
        5. 支持数据崩溃后的数据安全恢复
        6. InnoDB 的主键索引称之为聚簇索引，也就是说数据和索引是存储在一起的
        7. InnoDB 的辅助索引只存储索引值+主键ID
        8. 查询时若是辅助索引不能覆盖查询列时，就需要通过主键ID进行回表

        总结：

        ​ InnoDB之所以能成为数据库默认引擎，主要是对事务的支持、和崩溃后的数据安全恢复，比较出名的时MVCC、next-key lock、redolog、undolog；

        ​ 数据库事务通常也是借助日志来实现，常见的undo log、redo log，undo/redo log都能保证事务特性，这里主要是原子性和持久性，即事务相关的操作，要么全做，要么不做，并且修改的数据能得到持久化。

    3. 日志

      1. 二进制日志 BingLog
      2. 重做日志 Redolog
        1. 宕机恢复时，对于已提交的事务重新执行
      3. 回滚日志 UndoLog
        1. 宕机恢复时，对于未提交的事务进行回滚
      4. 还有慢查询日志、错误日志、查询日志等

    4. 索引

      1. 主键索引（聚簇索引）
      2. 非主键索引（辅助索引）
      3. 最左匹配
      4. 联合索引

      6. 索引下推

        2. 可以避免老版本（5.6之前先回表再过滤），查询左侧条件之后的回表操作（查询的结果都回表）

      7. 索引选择（优化器怎么选择索引）
        1. 参考《Mysql - 优化器是如何选择索引的？》
        2. 一条语句一般会命中多个索引
        3. 优化器的选择原则
          1. 在数据库里面，扫描行数是影响执行代价的因素之一。
          2. 扫描的行数越少，意味着访问磁盘数据的次数越少，消耗的 CPU 资源越少。
          3. 当然，扫描行数并不是唯一的判断标准，优化器还会结合是否使用临时表、是否排序等因素进行综合判断。
        4. 优化器如何判断扫描行数
          1. 统计索引的区分度
          2. 使用采样统计的方式，粗略的估计扫描行
    2. 为什么使用B+树而不使用B树
    3. 聚集索引与非聚集索引（从存储引擎上进行分析）
    4. 数据库为什么会默认创建主键索引
      1. InnoDB存储引擎的数据结构决定
      2. InnoDB存储引擎的主键索引和数据是存在一起的，使用B+树的数据结构
      3. 数据是存储在叶子节点的，所以InnoDB存储引擎必须得有主键
    5. Mysql主键设计（推荐使用自增Id）
      1. 使用自增Id，可以有序的在数据页尾部进行插入，页满的时候直接开辟新的数据页
      2. 无需移动数据
      3. 使用非自增Id的话（数据在数据页中非连续存储，插入数据的时候可能会插入到中间位置，导致页裂移动数据）

    Redis相关问题

    1. Redis的集群模式
    2. Redis的主从同步
    3. Redis做缓存时的数据一致性问题（如何保证redis缓存与数据一致性问题 - 简书 (jianshu.com)）
      1. 是先更新缓存中的数据再更新数据库的数据；
      2. 尽量保持数据的最终一致性吧
        1. 先更新数据库数据，再更新缓存数据
        2. 如果更新缓存失败了，不管原来的缓存存不存在，只要是到了过期时间，那么再次读操作的时候，就会从数据库中读取数据，然后更新到缓存中。
        3. 先删缓存，再更新数据库，需要考虑并发问题
          1. 这种处理方式，可能会有如下情况，线程A，写入(更新)数据时，先删除缓存后，同时，有一个线程B，此时去读数据，先去缓存中读，此时刚被线程A删除了，于是去数据库读还未被线程A更新的数据，这时候就会读到脏数据了。 紧接着，线程B会把脏数据写入缓存，而线程A会把更新后的数据写入数据库。导致缓存与数据不一致。
          2. 处理方式:
            1. 双删策略（先删缓存、更新数据库、休眠1s再次删除缓存（删除1s内产生的咱数据））
        4. 先更新数据，再更新缓存
          1. 线程A先执行，线程B后执行。线程A先查询时，去缓存读取时，缓存失效了，此时继续去数据库读，这时候线程B修改数据库后，把数据写入缓存了。在此之后，线程A才执行把数据写入缓存中，此时缓存中的任然是脏数据。
          2. 分为以下几个步骤:
            （1）缓存刚好失效
            （2）请求A查询数据库，得一个旧值
            （3）请求B将新值写入数据库
            （4）请求B删除缓存
            （5）请求A将查到的旧值写入缓存
          3. 如果一定需要做控制的话，可以考虑以下几点:
            1. 做redis过期策略。(时间上不好控制)
            2. 上面说的延迟删除策略。(删除不一定成功)
    4. Redis分布式锁
      1. 尽量使用Redission框架
      2. 自己实现分布式锁要考虑的点
        1. 互斥，同一时刻只能有一个线程持有锁
        2. 防止死锁，设置持有锁的过期时间内
        3. 可重入，同一线程可多次持有锁
        4. 锁续期，防止任务处理时间 > 锁过期时间
    5. Redis常见问题
      1. 雪崩
        1. 大量的key同时失效（原因很多），大量请求打到数据库上，可能导致数据库崩溃
        2. 大量key同时失效的原因
          1. redis服务挂掉
          2. 对缓存数据设置了相同的过期时间
        3. 解决方案：
          1. 搭建高可用的Redis集群（Redis Cluster 或者 Redis Sentinel(哨兵) 等方案）
          2. 缓存时间加上一个随机值，避免某一时间缓存集中失效
      2. 穿透
        1. 大量请求一个不存在的数据，Redis内存屏障失效，请求直接落到数据库上，导致数据库崩溃
        2. 解决方案：
          1. 使用布隆过滤器(Bloom Filter)过滤掉非法请求
          2. 缓存空数据
      3. 击穿
        1. 热点数据，在失效的那一瞬间，大量的请求打到数据库上
        2. 解决方案：
          1. 让热点Key永不过期
          2. 使用互斥锁，通过Redis的setnx实现（同一时刻只能有一个请求去请求数据库，等这个请求将mysql的内容加载到缓存，其他请求就能从缓存中获取数据了）
